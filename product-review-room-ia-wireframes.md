# 产品评审室 信息架构与页面原型说明

## 1. 文档信息

- 产品名称：产品评审室
- 文档类型：信息架构与页面原型说明
- 适用范围：MVP 网页端
- 文档日期：2026-03-31

## 2. 设计目标

本产品的网页端设计目标不是做成聊天应用，而是做成一个带有专业判断感的产品经理工作台。用户进入网站后，应感受到这是一个用于发起需求评审、沉淀干系人视角、整理评审结论的高密度工具，而不是陪聊型 AI 界面。

MVP 页面设计需优先满足以下目标：

1. 快速发起一次评审
2. 清楚展示上传材料与角色配置
3. 让 1 vs N 评审对话有秩序、可跟踪
4. 将角色资产和评审输出清晰沉淀

```text
├── 首页 / 工作台
├── 新建评审室

│   ├── 文件上传
│   ├── 角色选择 / 角色生成
│   └── 我的评审风格
├── 评审对话室

│   ├── 角色面板

│   ├── 我的角色
│   └── 角色详情
└── 历史评审详情

    ├── 自动总结
    └── 导出
```

## 4. 全局导航设计

### 4.1 导航结构

左侧主导航建议固定为：

1. 工作台
2. 新建评审室
3. 角色中心
4. 历史评审

右上角功能建议保留：

1. 全局搜索入口
2. 导出入口
3. 用户设置

### 4.2 导航原则

1. 全站层级控制在两层内
2. 高频操作直接暴露，不藏在二级菜单
3. 每个页面都能快速返回“新建评审室”

## 5. 页面一：首页 / 工作台

### 5.1 页面目标

帮助用户快速回到上次工作上下文，并尽快开始下一次评审。

### 5.2 页面布局

建议采用三段式布局：

1. 顶部：欢迎区 + 新建按钮
2. 中部：最近评审室
3. 下部：角色模板快捷区 + 最近导出

### 5.3 页面模块

#### 模块 A：欢迎区

内容：

- 页面标题：产品评审室
- 简短说明：上传需求材料，拉起虚拟干系人，提前完成一次需求评审
- 主按钮：`新建评审室`
- 次按钮：`进入角色中心`

#### 模块 B：最近评审室

展示形式：

- 卡片列表

字段建议：

- 评审室名称
- 主题
- 最近更新时间
- 参与角色数量
- 最近一次总结标签

卡片操作：

- 继续评审
- 查看详情
- 导出记录

#### 模块 C：常用角色模板

展示形式：

- 横向角色卡片

每张卡展示：

- 名称
- 生态位
- 关注重点关键词
- 最近使用次数

操作：

- 查看详情
- 用于新评审

#### 模块 D：最近导出

展示形式：

- 简表

字段：

- 评审室名称
- 导出时间
- 导出格式

### 5.4 页面原型描述

```text
+--------------------------------------------------------------+
| Logo | 工作台 | 新建评审室 | 角色中心 | 历史评审 | 搜索 | 用户 |

| 上传需求材料，拉起虚拟干系人，提前完成一次需求评审             |

+--------------------------------------------------------------+
| 最近评审室                                                   |
```

## 6. 页面二：新建评审室

### 6.1 页面目标

帮助用户在一个连续流程中完成评审室创建、文件上传、角色确认和发起评审。

### 6.2 页面结构

建议采用分步式页面，但尽量保持在同一屏流程中完成。

四步结构：

1. 基础信息
2. 上传材料
3. 角色配置
4. 我的评审风格

### 6.3 模块设计

#### 模块 A：基础信息

字段：

- 评审室名称
- 评审主题
- 需求类型标签，可选

说明文案：

- 例如：支付改版需求评审、会员中心新增权益模块评审

#### 模块 B：上传材料

上传区需支持拖拽与点击上传。

支持格式：

- PDF
- DOCX
- MD
- TXT

文件类型标签：

- PRD
- 背景材料
- 会议记录

上传完成后展示：

- 文件名
- 格式
- 文件类型
- 解析状态

会议记录上传后，需要展示一个“识别到说话人”的预览区域。

#### 模块 C：角色配置

分为左右两栏：

左侧：

- 预置角色模板列表
- 我的历史角色列表

右侧：

- 从会议记录生成的角色草稿
- 当前已选角色列表

角色卡片字段：

- 名称
- 生态位
- 核心利益
- 关注重点
- 表达风格

卡片操作：

- 添加到本次评审
- 编辑
- 删除

#### 模块 D：我的评审风格

形式建议：

- 轻量表单 + 标签选择

字段：

- 关注倾向
- 表达方式
- 提问习惯

页面文案建议：

- 这会影响系统如何理解你在评审中的提问方式与偏好

#### 模块 E：底部操作区

按钮：

- 保存草稿
- 开始评审

```text
+--------------------------------------------------------------+
```

## 7. 页面三：评审对话室

### 7.1 页面目标

让用户围绕一个明确主题，与多位虚拟干系人进行稳定、结构化的需求评审。

### 7.2 页面布局

建议采用三栏布局：

1. 左栏：评审背景与材料
2. 中栏：会话流
3. 右栏：角色面板与总结面板

### 7.3 模块设计

#### 模块 A：左栏 - 评审概览

内容：

- 评审室名称
- 当前主题
- 上传材料列表
- 参与角色列表

交互：

- 点击文件可查看摘要
- 点击角色可查看角色卡详情

#### 模块 B：中栏 - 会话流

消息类型：

- 用户发言
- 角色发言
- 系统提示
- 阶段总结

每条角色消息显示：

- 角色名称
- 生态位标签
- 发言时间
- 发言内容

输入区：

- 文本输入框
- 发送按钮
- 可选快捷指令按钮，如“请指出最大风险”“请分别给出是否支持结论”

顺序机制：

- 用户发问后，角色按既定顺序依次发言
- 当前发言角色高亮
- 本轮完成后显示“下一轮继续追问”

#### 模块 C：右栏 - 角色面板

每个角色展示：

- 名称
- 生态位
- 核心利益
- 关注重点
- 当前态度：支持 / 保留 / 反对

用途：

- 让用户在对话中始终看见“谁站在什么立场说话”

#### 模块 D：右栏 - 自动总结

总结模块需固定可见或可折叠。

内容：

- 主要共识
- 主要分歧
- 关键风险
- 建议下一步

按钮：

- 复制总结
- 导出记录

### 7.4 页面原型描述

```text
+------------------+--------------------------------+------------------+
| 评审概览         | 会话流                         | 角色 / 总结      |
| 主题             | 用户：这个需求的目标是...      | 角色1：研发负责人 |
| 上传材料         | 系统：开始第一轮评审           | 生态位：交付 owner|
| PRD.pdf          | 角色A：我最大的担忧是...       | 利益：稳定交付     |
| 会议记录.txt     | 角色B：从运营角度看...         | 重点：成本/周期    |
|                  | 角色C：用户价值不够明确        | 当前态度：保留     |
| 参与角色         |                                |------------------|
| 研发 运营 用户   | [输入框__________________]发送 | 主要共识          |
|                  |                                | 主要分歧          |
|                  |                                | 关键风险          |
|                  |                                | 建议下一步        |
|                  |                                | [导出记录]        |
+------------------+--------------------------------+------------------+
```

## 10. 核心交互流程

### 10.1 流程一：从零创建评审室

1. 进入工作台
2. 点击“新建评审室”
3. 填写名称与主题
4. 上传 PRD、背景材料、会议记录
5. 系统完成解析
6. 用户选择预置角色，或从会议记录生成角色
7. 用户设置“我的评审风格”
8. 点击“开始评审”
9. 进入评审对话室

### 10.2 流程二：从会议记录一键生成角色

1. 上传会议记录
2. 系统识别说话人
3. 展示说话人列表与角色草稿
4. 用户逐个确认或编辑
5. 保存到当前评审室
6. 可选择存入角色中心

### 10.3 流程三：发起 1 vs N 评审

1. 用户在输入框中提出需求问题
2. 系统显示“开始第一轮评审”
3. 角色按顺序发言
4. 用户继续追问
5. 第二轮角色继续回应
6. 系统生成总结
7. 用户导出记录

## 12. 异常与空状态设计

### 12.1 首页空状态

文案建议：

- 还没有评审室。上传一份 PRD，开始你的第一次需求评审。

按钮：

- 新建评审室

### 12.2 无会议记录时的角色区空状态

文案建议：

- 还没有从会议记录生成角色。你可以先使用预置角色模板，或上传带说话人的会议记录。

### 12.3 文件解析失败

文案建议：

- 这份文件暂时无法稳定解析。建议检查格式，或重新上传文本版内容。

### 12.4 无总结时

文案建议：

- 完成至少一轮评审后，这里会生成共识、分歧和风险摘要。

## 13. 视觉与体验方向

### 13.1 整体风格

建议采用“专业评审台”风格，而非常见 AI 紫色聊天界面。

方向建议：

- 偏浅色底
- 低饱和中性色为主
- 用一到两种有判断感的强调色，如深蓝、墨绿、锈红
- 字体风格偏编辑感和工具感结合

### 13.2 视觉重点

1. 角色卡片要有明显识别感
2. 总结面板要稳定、可信、可复制
3. 文件与会话内容要有专业文档感
4. 让“评审”而不是“聊天”成为视觉中心
