# 产品需求文档 PRD

## 1. 文档信息

- 产品名称：产品评审室
- 产品定位：面向个人产品经理的 AI 需求评审工作台
- 文档版本：V1.0 MVP Draft
- 文档日期：2026-03-31
- 文档状态：待评审

## 2. 项目背景

个人产品经理在推进需求时，常常需要同时扮演多个角色：提出方案的人、与研发沟通的人、平衡商业目标的人、理解用户的人、预判风险的人。真实工作中，这些声音来自老板、研发、设计、运营、销售、客服、法务、用户等不同干系人，但个人产品经理往往无法在需求早期低成本地反复进行高质量评审。

现有 AI 产品大多提供单模型问答或娱乐型多角色对话，缺少针对需求评审场景的结构化方法、干系人角色沉淀机制，以及围绕 PRD、背景材料、会议纪要进行的真实评审能力。

因此，产品评审室希望为个人产品经理提供一个可反复使用的虚拟评审环境，帮助其围绕具体需求发起 1 vs N 的需求评审，并持续沉淀角色画像与评审资产。

## 3. 产品目标

### 3.1 目标

构建一款面向个人产品经理的网页端工具，使用户能够：

1. 上传 PRD、背景文档、会议纪要等材料，快速发起一次需求评审。
2. 使用预置角色模板，或基于会议记录和历史对话生成、修正角色。
3. 让多个虚拟干系人围绕同一需求进行结构化讨论。
4. 沉淀角色的生态位、核心利益和关注重点，为后续评审复用。
5. 导出评审会话记录，作为后续整理和推进的基础材料。

### 3.2 非目标

以下内容不属于 MVP 范围：

1. 多人在线协作与共享评审室
2. 音视频转写、原始录音上传与自动识别
3. 用户中途加入对话并重新编排讨论流程
4. Jira、飞书任务系统等第三方任务同步
5. 完整知识库平台能力与跨项目长期文档治理

## 4. 目标用户

### 4.1 核心用户

- 个人产品经理

### 4.2 用户特征

- 需要频繁撰写或评审需求
- 希望在正式开评审会前进行预演
- 缺少随时可得的多角色反馈环境
- 需要保留不同干系人的典型视角，避免每次从零开始 prompt

### 4.3 核心使用场景

1. 产品经理完成 PRD 初稿后，上传文档进行预评审。
2. 产品经理导入历史会议纪要，从真实说话人中提炼角色。
3. 产品经理在需求推进中，快速拉起某类干系人组合进行意见碰撞。
4. 产品经理在会后导出评审记录，整理为需求修订依据。

## 5. 核心价值主张

产品评审室为个人产品经理提供三类核心价值：

1. 低成本模拟真实评审
在正式评审前，用虚拟干系人提前暴露争议、风险和信息缺口。

2. 沉淀可复用的干系人资产
将角色从一次性 prompt 升级为可持续使用的干系人模板，长期保留其生态位、利益点和关注重点。

3. 让评审过程结构化
通过固定的 1 vs N 评审流程，把“讨论”转化为可重复执行的方法，而不是一次性聊天。

## 6. 产品定义

产品评审室是一个面向个人产品经理的 AI 需求评审网页端。用户以“产品经理本人”的身份创建评审室，上传 PRD、背景资料和会议纪要，选择或生成干系人角色，由系统组织多个角色围绕一个需求进行有顺序的评审发言，并将会话与角色信息沉淀为后续可复用资产。

## 7. MVP 范围

### 7.1 包含内容

1. 新建评审室
2. 上传 PRD、背景文档、会议记录
3. 解析带说话人的会议记录
4. 基于会议记录一键创建角色
5. 提供预置角色模板并允许修改
6. 支持设置“我”的评审风格
7. 发起 1 vs N 需求评审
8. 按顺序轮流发言的评审对话
9. 自动生成简要总结
10. 导出评审会话记录

### 7.2 暂不包含

1. 多用户系统
2. 实时协同编辑
3. 自由乱序群聊
4. 录音转写
5. 自动生成完整 PRD
6. 第三方工具集成

## 8. 用户故事

1. 作为产品经理，我希望上传 PRD 和背景材料后快速创建评审室，这样我可以在正式开会前做一次预评审。
2. 作为产品经理，我希望导入已标注说话人的会议纪要，并一键生成角色，这样我可以让 AI 更像我熟悉的真实干系人。
3. 作为产品经理，我希望系统内置常见干系人模板，同时允许我调整角色设定，这样角色能更贴近我的组织环境。
4. 作为产品经理，我希望以“我 vs 多位评审角色”的方式推进讨论，这样更接近真实需求评审过程。
5. 作为产品经理，我希望每位角色有明确的利益诉求和关注重点，这样意见不会流于空泛。
6. 作为产品经理，我希望评审结束后能导出会话记录，这样我可以继续整理和推进需求。

## 9. 关键概念定义

### 9.1 评审室

围绕某一具体需求创建的独立工作空间，包含主题、上传文件、参与角色、会话记录和输出结果。

### 9.2 角色

用于模拟真实干系人视角的 AI 实体。角色由以下信息组成：

- 名称
- 生态位
- 核心利益
- 关注重点
- 典型表达风格
- 风格来源
- 是否为强势反对型、审慎型、支持型等互动倾向

### 9.3 我的评审风格

用户作为产品经理本人的固定设定，用于影响提问方式、追问方式和对讨论的偏好，例如更关注增长、偏向风险控制、表达更直接等。

### 9.4 文件解析

对上传的 PRD、背景材料、会议纪要进行文本提取、结构识别和角色特征抽取的能力。

## 10. 核心流程

### 10.1 主流程

1. 用户创建评审室
2. 用户上传文件
3. 系统解析文件内容
4. 用户选择预置角色或从会议记录生成角色
5. 用户确认或编辑角色设定
6. 用户设置“我”的评审风格
7. 系统按顺序组织 1 vs N 需求评审
8. 系统输出会话内容与简要总结
9. 用户导出会话记录

### 10.2 文件与角色生成流程

1. 用户上传会议记录
2. 系统识别说话人及发言片段
3. 系统聚合同一说话人的表达特征
4. 系统提取该说话人的生态位、核心利益、关注重点和语言风格
5. 系统生成角色草稿
6. 用户确认或修改后保存为角色

### 10.3 对话流程

1. 用户输入评审主题或选择上传的需求作为主题
2. 用户发起第一轮问题
3. 系统按照预设顺序让角色逐个发言
4. 用户可在轮次之间继续提问或补充上下文
5. 系统在当前轮次结束后生成阶段性小结
6. 会话结束后生成总体总结

## 11. 功能需求

### 11.1 新建评审室

功能描述：
用户创建一场新的需求评审会话。

功能要求：

1. 支持填写评审室名称
2. 支持填写评审主题
3. 支持上传相关文件
4. 支持选择已有角色模板
5. 支持从上传文件中生成角色

输入：

- 评审室名称
- 评审主题
- 文件

输出：

- 新的评审室记录

### 11.2 文件上传与解析

功能描述：
支持用户在创建评审室时上传文件，并从文件中提取评审上下文。

支持格式：

- PDF
- DOCX
- MD
- TXT

功能要求：

1. 支持单个和多个文件上传
2. 支持将文件标记为 PRD、背景材料、会议记录
3. 对文本文件进行内容抽取
4. 对会议记录识别说话人分段
5. 为后续角色生成和会话检索提供上下文

约束：

1. 第一版仅支持已整理好的、带说话人的会议记录文本
2. 不支持音频直接转写

### 11.3 角色模板中心

功能描述：
提供预置角色模板，并支持用户修改与复用。

预置角色示例：

- 研发负责人
- 设计负责人
- 运营负责人
- 销售代表
- 客服代表
- 老板
- 典型用户

模板字段：

- 名称
- 生态位
- 核心利益
- 关注重点
- 典型表达风格
- 互动倾向

功能要求：

1. 支持查看预置模板
2. 支持复制预置模板并编辑
3. 支持保存为个人模板
4. 支持在新评审室中复用历史角色

### 11.4 从会议记录生成角色

功能描述：
从已上传的带说话人会议记录中，一键生成角色草稿。

功能要求：

1. 识别说话人名称
2. 提取其典型话语片段
3. 推断其生态位
4. 推断其核心利益
5. 推断其关注重点
6. 生成典型表达风格标签
7. 允许用户逐项编辑

输出字段：

- 角色名称
- 角色生态位
- 核心利益
- 关注重点
- 典型表达风格
- 互动倾向
- 风格来源摘要

### 11.5 我的评审风格

功能描述：
允许用户为“产品经理本人”配置一个基础风格。

可配置项：

- 更关注增长 / 更关注体验 / 更关注风险 / 更关注效率
- 表达直接或克制
- 习惯先问目标还是先问方案

功能要求：

1. 支持首次设置
2. 支持后续修改
3. 在评审发问、总结和系统引导中生效

### 11.6 1 vs N 需求评审对话

功能描述：
用户作为“1”，与 N 位评审角色展开结构化对话。

机制要求：

1. N 可为 1 人或多人
2. 角色按顺序轮流发言
3. 用户可在轮次之间继续追问
4. 发言需结合上传文件内容
5. 每位角色应围绕自身利益和关注点表达意见

第一版建议轮次结构：

1. 用户提出需求背景和问题
2. 角色逐一发表首轮意见
3. 用户追问争议点
4. 角色逐一补充或反驳
5. 系统输出总结

### 11.7 自动总结

功能描述：
在评审结束后自动生成简短总结，帮助用户快速回顾。

总结内容：

- 主要共识
- 主要分歧
- 关键风险
- 建议下一步

### 11.8 会话记录导出

功能描述：
支持导出本次评审会话记录。

第一版导出格式：

- Markdown
- PDF

导出内容：

- 评审室名称
- 评审主题
- 参与角色
- 评审时间
- 完整会话记录
- 自动总结

## 13. 交互原则

1. 强调工作台感而非娱乐聊天室感
2. 先上传材料，再开始讨论，避免角色空谈
3. 角色设定尽量结构化，减少用户从零写 prompt 的负担
4. 总结与导出要一键可得
5. 会话中角色立场要稳定，不应频繁漂移

## 14. AI 能力设计

### 14.1 角色生成机制

输入：

- 会议纪要文本
- 预置角色模板
- 用户编辑内容

处理：

1. 识别说话人
2. 聚合说话风格
3. 抽取角色特征
4. 生成角色草稿

输出：

- 可编辑角色对象

### 14.2 对话生成机制

输入：

- 评审主题
- 上传文件内容
- 用户本人风格
- 角色设定

处理：

1. 检索相关文档片段
2. 将片段与角色设定拼装为上下文
3. 按角色顺序生成发言
4. 控制发言围绕自身核心利益与关注点
5. 生成阶段总结与最终总结

### 14.3 知识使用机制

第一版知识来源限制在当前评审室上传文件，不做跨项目广域知识库。

机制要求：

1. 文件切片与向量化
2. 基于当前问题进行检索
3. 角色发言优先引用相关文件内容
4. 当文档无依据时，可表达假设，但需降低确定性

## 17. 成功指标

MVP 阶段建议观察以下指标：

1. 创建评审室完成率
2. 上传文件后成功进入评审的比例
3. 从会议记录成功生成角色的比例
4. 单次评审平均对话轮数
5. 会话导出使用率
6. 角色复用率

## 18. 风险与难点

1. 会议记录质量参差不齐，可能导致角色抽取失真
2. 角色如果不稳定，用户会觉得“不像真人”
3. 如果检索不到有效上下文，角色容易空谈
4. 多角色对话成本较高，需控制轮次和上下文长度
5. 如果总结泛泛而谈，会削弱产品可信度

## 19. MVP 验收标准

1. 用户可新建评审室并上传支持格式文件
2. 系统可解析带说话人的会议记录
3. 用户可查看并编辑系统生成的角色
4. 用户可选择多个角色发起 1 vs N 需求评审
5. 对话支持按顺序轮流发言
6. 系统可生成会话总结
7. 用户可导出 Markdown 和 PDF 格式的会话记录

## 20. 后续迭代方向

1. 角色关系网络与推荐组合
2. 更多评审模板，如方案评审、复盘评审
3. 更强的角色学习机制，持续从历史会话中修正角色
4. 支持生成需求修订建议
5. 支持导出为 PRD 修订稿或评审纪要
6. 团队版与共享评审室
